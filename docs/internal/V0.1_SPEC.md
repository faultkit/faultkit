# faultkit v0.1 implementation spec

**Audience:** Claude Code, working on `github.com/faultkit-dev/faultkit`.
**Status:** active. This document supplements `PROJECT_PLAN.md` and `CLAUDE.md`.
**Scope:** v0.1 only. Anything not listed here is out of scope for this iteration.

If something here conflicts with `CLAUDE.md`, `CLAUDE.md` wins. If something
here conflicts with `PROJECT_PLAN.md`, this document wins (it represents the
current iteration of the plan after a strategic refinement toward AI-agent-first
positioning).

---

## 1. The strategic shift, in one paragraph

The original plan positioned faultkit as an eBPF-first chaos engineering tool
with AI scenarios as a wedge. After review, the positioning has been refined:
faultkit is **a fault injection toolkit for AI/agent stacks**, where the
HTTPS proxy is the primary injection mechanism (because LLM/RAG/gateway
failures live in HTTPS semantics on top of TLS), and eBPF is the supporting
mechanism for syscall-level scenarios (backend chaos, future tool-call
scenarios). v0.1 must ship both injection paths working end-to-end, but the
marquee scenario, the integration test, and the launch story are all
proxy-and-AI-agent shaped.

---

## 2. v0.1 deliverables

v0.1 ships when all of the following are true:

1. `faultkit run --scenario llm-api-degraded -- pytest tests/agent/` works on
   macOS and Linux against real `openai-python` SDK calls. The proxy returns
   real HTTP 429s with vendor-accurate response bodies. The target test exits
   non-zero when the agent's retry path is broken.
2. `faultkit run --scenario flaky-network -- <some-target>` works on Linux
   5.8+ and demonstrates eBPF-mode injection with a real `ECONNRESET`. This
   is the "yes, both modes work" proof point.
3. `faultkit doctor` reports which modes are available on the current host
   and why. Honest, no false positives.
4. Distinct exit codes per `CLAUDE.md` (`0` = pass, `1` = target failed
   under fault, `2` = internal error, `3` = fault never fired, `4` = usage).
5. JSON report output via `--report report.json` for CI consumption.
6. GitHub Actions example in `examples/github-actions/` that runs against
   a tiny test target and demonstrates the "fault catches a bug" flow.
7. `make build`, `make test`, `make lint` all pass in CI.
8. README from this conversation is in place. PROJECT_PLAN.md is as-is.
9. Apache 2.0 LICENSE file. CONTRIBUTING.md with CLA bot pointer (CLA setup
   itself is out of scope; the doc just references it). CODE_OF_CONDUCT.md.

What is **not** in v0.1 (do not build, do not stub, do not scaffold):

- Anthropic, Bedrock, Vertex, or any non-OpenAI provider fixtures
- Streaming/SSE handling (synchronous JSON only)
- LD_PRELOAD / DYLD shim mode
- More than one eBPF scenario
- `tool-call-flaky`, `rag-corruption`, `context-window-squeeze`, or any
  scenario other than the two named in deliverables 1 and 2
- iptables-based redirection (only env-var-based proxy redirection in v0.1)
- BPF cgroup connect hook (deferred to v0.2 when env-var-based is found
  to be insufficient)
- Pro-aware code of any kind
- Telemetry, analytics, version checks, "phone home"
- Auto-update logic
- Configuration via anything other than YAML files + CLI flags

If a request leans into any of the above, stop and surface it.

## 3. Architecture decisions, locked

These are not up for re-litigation in implementation. If implementation
reveals a decision is wrong, stop and ask before changing it.

### 3.1 Languages and runtimes

- **CLI, runner, proxy, scenario engine, eBPF loader: Go.** Single binary
  output. `go.mod` Go version: 1.22 minimum.
- **eBPF programs: C with libbpf-style CO-RE.** GPL-2.0 licensed (kernel
  requirement). Compiled with `clang` to `bpf` target, embedded with
  `go:embed` so the released binary needs no clang at runtime.
- **Shim: C.** Not in v0.1, but the directory `shim/` is created empty
  with a placeholder README explaining v0.2 status.

### 3.2 Injection mechanisms

| Mechanism    | Status in v0.1 | Used for                             |
|--------------|----------------|--------------------------------------|
| HTTPS proxy  | shipped        | `llm-api-degraded` (and future LLM/RAG/gateway scenarios) |
| eBPF         | shipped, minimal | `flaky-network` (and future syscall-level scenarios)    |
| Shim         | not built      | placeholder dir only                  |

The proxy uses **martian v3** (`github.com/google/martian/v3`). This is the
only new dependency v0.1 may add. All other proxy work builds on stdlib
+ existing dependencies in `go.mod`.

### 3.3 Proxy redirection: env-var-only in v0.1

v0.1 redirects the target's HTTPS traffic to the proxy via environment
variables (`HTTPS_PROXY`, `HTTP_PROXY`, lowercase variants) and trusts the
per-run CA via `SSL_CERT_FILE`, `REQUESTS_CA_BUNDLE`, `NODE_EXTRA_CA_CERTS`,
`CURL_CA_BUNDLE`. This catches the entire Python/Node OpenAI ecosystem,
which is the v0.1 audience. Clients that ignore `HTTPS_PROXY` (some Go
configurations, some Rust `reqwest` setups) are documented as a known
limitation; eBPF-based redirection (`BPF_PROG_TYPE_CGROUP_SOCK_ADDR`) is
deferred to v0.2.

### 3.4 TLS: per-run ephemeral CA

A fresh CA is generated on every `faultkit run` invocation, written to a
temp file, used for the run, and deleted on `Stop`. Never cache. Never
write to a stable path. Never install in any system trust store. The CA's
`NotAfter` is 24 hours, but the run is expected to clean up well before
that.

### 3.5 OpenAI-only fixtures in v0.1

Vendor-accurate response bodies for OpenAI only. Other vendors fall through
to a generic shape:

```json
{"error": {"message": "...", "type": "rate_limit_error", "code": "rate_limit_exceeded"}}
```

Adding a vendor fixture is a public API commitment per `CLAUDE.md` rules.
Do not add new vendor fixtures in v0.1, even if asked.

### 3.6 The `x-faultkit-synthetic` header

Every synthetic response includes `x-faultkit-synthetic: true`. This is a
deliberate choice: it weakens the "indistinguishable from production"
pitch slightly, but it preserves debuggability and prevents synthetic
responses from being misread by production telemetry if something leaks.
Production agent code does not and should not read this header. It is
human-facing metadata.

### 3.7 No fault decision in eBPF for v0.1

The eBPF program in v0.1 is dumb: it observes syscalls and reports them
to userspace via a perf buffer. Fault decisions (probability, matching,
timing) live in Go. This is more verifier-friendly, easier to test, and
keeps the BPF code small. Fault injection itself uses
`bpf_override_return` based on a userspace-populated map. Specifically:

- BPF program checks an LRU hashmap keyed by `(pid, syscall_id, fault_token)`
- Userspace populates the map ahead of time based on the scenario
- BPF reads the map; if entry exists and matches, calls `bpf_override_return(-ECONNRESET)`
- All other logic (probability, host matching, etc.) is in Go

### 3.8 Single-shot, foreground only

faultkit runs as `faultkit run -- <target>`. It forks the target, waits
for it to exit, cleans up, and exits with the appropriate code. There is
no daemon mode, no background mode, no `serve` subcommand. This is the
contract from `PROJECT_PLAN.md` and stands.

## 5. Implementation phases

Each phase has explicit success criteria. Do not start phase N+1 until
phase N's criteria are met. If a criterion is unclear, surface it and ask.

### Phase 0 — Repo hygiene (~half a weekend)

**Tasks:**
- Initialize the repo structure above (empty/placeholder files where needed).
- `go.mod` with declared dependencies: `github.com/cilium/ebpf`,
  `github.com/spf13/cobra`, `github.com/spf13/viper`, `gopkg.in/yaml.v3`,
  `github.com/google/martian/v3`. No others.
- `Makefile` with `build`, `test`, `lint`, `bpf`, `clean` targets.
- `.golangci.yml` with reasonable defaults (`govet`, `staticcheck`,
  `errcheck`, `gofmt`, `goimports`, `gosec`, `revive`).
- LICENSE (Apache 2.0), CONTRIBUTING.md (skeleton), CODE_OF_CONDUCT.md
  (Contributor Covenant 2.1).
- `.github/workflows/ci.yml` running `make lint test` on Linux + macOS.
- Initial commit per `CLAUDE.md` commit conventions.

**Success criteria:**
- `make build` succeeds and produces `bin/faultkit`.
- `bin/faultkit version` prints a version string.
- `make lint test` passes in CI on a fresh clone.

### Phase 1 — Scenario loading and CLI plumbing (~half a weekend)

**Tasks:**
- Define `pkg/scenario.Scenario` and `pkg/scenario.Experiment` structs.
  YAML schema below in §6.
- Implement `pkg/scenario.Load(path string) (*Scenario, error)` and
  `pkg/scenario.LoadBuiltin(name string) (*Scenario, error)`.
- Implement `internal/scenario/builtin/` with two builtin scenarios:
  `llm-api-degraded` and `flaky-network`. Embed their YAML with
  `go:embed`. Register via `init()` per `CLAUDE.md` conventions.
- Implement `cmd/faultkit/main.go` (≤20 LOC of substance, just calls
  `internal/cli.Execute()`).
- Implement `internal/cli/cli.go` with cobra root command and exit code
  dispatch. Define typed exit code constants (`ExitOK`, `ExitTargetFailed`,
  `ExitInternalError`, `ExitFaultNotFired`, `ExitUsage`).
- Implement `faultkit version`, `faultkit scenario list`,
  `faultkit scenario show <name>` subcommands. These don't depend on
  injection and can be built first as smoke tests.
- Implement skeleton `faultkit run` and `faultkit doctor` (will be filled
  in later phases).

**Success criteria:**
- `faultkit scenario list` prints both builtin scenarios with one-line
  descriptions.
- `faultkit scenario show llm-api-degraded` prints the loaded YAML.
- Loading a malformed YAML fails with a useful error and exit code 4.
- `faultkit version` prints version + commit + build date.
- Unit tests for scenario loading (table-driven, valid + invalid YAMLs).

### Phase 2 — Per-run CA and proxy server (~1 weekend)

**Tasks:**
- Implement `internal/inject/proxy/ca.go`:
  - `NewCA()` generates a per-run ECDSA P-256 root.
  - `MintLeaf(host string)` returns a `*tls.Certificate` for `host`,
    cached per host, with appropriate `DNSNames` SAN.
  - `WriteCertPEM()` writes the CA cert to a temp file and returns the
    path. Tracks the path for cleanup.
  - `Cleanup()` removes the temp file. Idempotent.
- Implement `internal/inject/proxy/server.go`:
  - Constructs a `martian.Proxy` configured with the per-run CA via
    `mitm.NewConfig(ca.cert, ca.key)`.
  - Sets request and response modifiers (the `Faulter`, see Phase 3).
  - Wraps the proxy in an `*http.Server` with sensible timeouts that
    don't break long-running streams (no `ReadTimeout`; bounded
    `ReadHeaderTimeout` only).
- Implement `internal/inject/proxy/proxy.go`:
  - `Injector` struct implementing `inject.Injector`.
  - `Start(ctx, scenario)` generates CA, listens on `127.0.0.1:0`,
    starts the proxy, returns env vars to inject into the target.
  - `Stop(ctx)` shuts down the proxy and cleans up the CA.
  - `Events()` returns the buffered event channel.

**Success criteria:**
- Unit tests for CA generation and leaf minting.
- A manual smoke test where the developer runs the proxy in a terminal
  and points `curl --cacert /tmp/faultkit-ca-*.pem -x http://127.0.0.1:8080
  https://api.openai.com/v1/models` at it and gets a real upstream response.
- Proxy doesn't leak the CA temp file after `Stop()`.

### Phase 3 — Fault matching and injection (~half a weekend)

**Tasks:**
- Implement `internal/inject/proxy/matcher.go`:
  - `Match(*http.Request) *Experiment` returns the first experiment whose
    `match` clause fits, in YAML order.
  - Host matching via `filepath.Match` (supports `*.openai.com`).
  - Path matching same.
- Implement `internal/inject/proxy/faulter.go`:
  - `ModifyRequest`: matches request → checks probability → if firing,
    marks context with `martian.Context.SkipRoundTrip()` and stashes
    the experiment for `ModifyResponse`.
  - `ModifyResponse`: reads stashed experiment, replaces response with
    synthetic one from `fixtures.SyntheticResponse`.
  - Emits `inject.Event` to the event channel for every fault fired.
- Implement `internal/inject/proxy/fixtures/openai.go`:
  - `OpenAIResponse(fault Fault) (body io.ReadCloser, headers map[string]string)`.
  - Vendor-accurate JSON shape per OpenAI's docs.
  - Sets `x-faultkit-synthetic: true`.
  - For 429: sets `Retry-After` from `fault.ResponseHeaders` or default 30.
- Implement `internal/inject/proxy/fixtures/fixtures.go`:
  - `SyntheticResponse(host string, fault Fault)` dispatches to the
    right vendor based on host. Default: generic shape.

**Success criteria:**
- Unit tests for matcher (table-driven, host glob patterns).
- Unit tests for faulter probability gating (deterministic via seeded RNG).
- Unit test for OpenAI fixture: response body matches OpenAI's documented
  rate limit error shape exactly.

### Phase 4 — Runner and target process management (~half a weekend)

**Tasks:**
- Implement `internal/runner/runner.go`:
  - `Runner.Run(ctx, target []string, env []string) (*Result, error)`.
  - Forks the target with merged env (target's env + faultkit-injected env).
  - Pipes stdout/stderr through to faultkit's stdout/stderr (no buffering;
    test output should stream live).
  - Waits for the target to exit. Captures exit code.
  - Propagates SIGINT/SIGTERM from faultkit to the target.
- Implement `internal/runner/errors.go`:
  - `*TargetExitError` typed error containing the target's non-zero
    exit code. The CLI dispatches this to `ExitTargetFailed`.

**Success criteria:**
- Test against a tiny shell script target: `runner.Run("sh", "-c", "exit 7")`
  returns `*TargetExitError` with `ExitCode == 7`.
- Test that env vars passed to `Run` reach the target (target prints
  `os.Getenv("HTTPS_PROXY")`, faultkit asserts on stdout).
- SIGINT to faultkit kills the target and faultkit exits cleanly.

### Phase 5 — `faultkit run` end-to-end with proxy (~half a weekend)

**Tasks:**
- Wire `internal/cli/run.go`:
  1. Parse flags (`--scenario`, `--config`, `--report`, `--mode`, `--`).
  2. Load scenario.
  3. Resolve injection mode (Phase 6 handles this, but for v0.1 with
     only proxy + ebpf, default selection: if scenario has any HTTP
     experiments → proxy; otherwise → ebpf).
  4. Construct injector (proxy in this phase).
  5. `injector.Start(ctx, scenario)` → get env.
  6. `runner.Run(ctx, target, env)` → get result.
  7. `injector.Stop(ctx)`.
  8. Drain events channel into report.
  9. Write report (terminal always, JSON if `--report`).
  10. Exit with appropriate code.
- Implement `internal/report/terminal.go` and `internal/report/json.go`.

**Success criteria:**
- `faultkit run --scenario llm-api-degraded -- curl https://api.openai.com/v1/models`
  returns a synthetic 429 some percentage of the time, real upstream the rest.
- `faultkit run --scenario llm-api-degraded -- false` exits with code 1
  and reports "target failed under fault."
- `faultkit run --scenario llm-api-degraded -- true` exits with code 3
  if the target made no HTTPS calls (fault never fired).
- `--report report.json` produces machine-parseable output.

### Phase 6 — `faultkit doctor` and mode resolution (~quarter weekend)

**Tasks:**
- Implement `internal/inject/inject_linux.go` and `inject_other.go`:
  - `AvailableModes() []Mode` returns which modes can run on this host.
  - On Linux: check kernel version (5.8+), check `CAP_BPF` /
    `CAP_NET_ADMIN` / root (for eBPF mode), report proxy as always
    available.
  - On non-Linux: proxy always available, eBPF reports unsupported.
- Implement `internal/cli/doctor.go`:
  - Prints platform, kernel version, available modes, why each mode is
    or isn't available.
  - Exits 0 if at least one mode works, 2 otherwise.

**Success criteria:**
- `faultkit doctor` on macOS reports "proxy: ok, ebpf: not supported (not Linux)".
- `faultkit doctor` on Linux without `CAP_BPF` reports "proxy: ok,
  ebpf: needs CAP_BPF + CAP_NET_ADMIN or root".
- `faultkit doctor` on Linux with the right caps reports both modes ok.

### Phase 7 — eBPF mode, single scenario (~1.5 weekends)

**Tasks:**
- Write `bpf/flaky_network.bpf.c`:
  - Hooks `tracepoint/syscalls/sys_enter_recvmsg` and the corresponding
    `bpf_override_return` site.
  - Reads from a fault-config map keyed by PID; if a fault is configured
    and the dice roll passes, calls `bpf_override_return(-ECONNRESET)`.
  - GPL-2.0 SPDX header.
  - Uses CO-RE for any struct field reads.
- Generate Go bindings via `bpf2go` (commit the generated files per
  `PROJECT_PLAN.md` so end users don't need clang).
- Implement `internal/inject/ebpf/ebpf_linux.go`:
  - `Injector` implementing `inject.Injector`.
  - `Start` loads the BPF program, attaches the tracepoint, populates
    the config map for the scenario's target PID tree (set after the
    runner forks — this means the eBPF Injector needs a hook to receive
    the target PID; thread this through the runner cleanly).
  - `Stop` detaches and unloads.
- Implement `internal/inject/ebpf/ebpf_stub.go` for non-Linux builds
  (returns `ErrUnsupported` from `Start`).

**Success criteria:**
- `faultkit run --scenario flaky-network -- curl http://example.com`
  on Linux occasionally fails with `Recv failure: Connection reset by peer`.
- The same command on macOS errors out cleanly with "eBPF mode not
  available on this platform; this scenario requires Linux 5.8+."
- An integration test confirms the fault rate is roughly the configured
  probability (statistical, not exact — assert within ±10%).

### Phase 8 — Integration test against real openai-python (~half a weekend)

**Tasks:**
- Implement `test/integration/proxy_openai_test.go`:
  - Uses `//go:build integration` tag.
  - Sets up: a fake OpenAI-compatible upstream (httptest.Server) returning
    a normal 200 response on `/v1/chat/completions`.
  - Runs `faultkit run --scenario llm-api-degraded -- python tests/agent.py`
    where the agent calls `openai.OpenAI(base_url=...).chat.completions.create(...)`.
  - Configures the scenario to match `match.host: <fake-upstream-host>`
    so the proxy intercepts, with `probability: 1.0` (deterministic).
  - Asserts: target exits non-zero, faultkit exits 1, JSON report shows
    the fault fired, the target's stderr contains the OpenAI SDK's
    rate-limit error message.
- Skip if Python or `openai-python` not available; document in the test.

**Success criteria:**
- `make test-integration` (a new make target) passes locally with Python
  + `openai` installed.
- The test fails informatively if `openai-python` produces an unexpected
  error (so future SDK upgrades surface as test failures, not silent passes).

### Phase 9 — Examples, docs, GitHub Actions (~half a weekend)

**Tasks:**

**Success criteria:**

## 6. Scenario YAML schema for v0.1

This is the v0.1 schema. Locked. Additions in v0.2+.

```yaml
# faultkit.yaml — top-level
name: my-scenario              # required, kebab-case
description: |                  # optional, one-line for `scenario list`
  One-line description of what this scenario tests.

# Optional. If absent, no constraint. If present, must match for the
# scenario to be loadable on this platform.
requires:
  platform: linux              # one of: linux, darwin, any
  kernel_min: "5.8"            # only if platform=linux

# One or more experiments. Each is independently evaluated per request
# (proxy mode) or per syscall (ebpf mode). First match wins.
experiments:
  - name: "openai rate-limited"
    fault:
      http_status: 429         # for proxy mode
      response_headers:        # optional, per-vendor sane defaults if absent
        Retry-After: "30"
    match:
      host: "api.openai.com"   # supports filepath.Match globs
      path: "/v1/*"            # optional, defaults to all paths
    probability: 0.2           # 0.0-1.0; 0.2 = 20% of matching requests
```

For the `flaky-network` builtin (eBPF mode):

```yaml
name: flaky-network
description: Inject ECONNRESET on TCP recv, simulating flaky network conditions.
requires:
  platform: linux
  kernel_min: "5.8"
experiments:
  - name: "tcp recv reset"
    fault:
      errno: ECONNRESET        # for ebpf mode
    match:
      syscall: recvmsg         # ebpf mode only
    probability: 0.1
```

The `Fault` struct in `pkg/faulttypes/`:

```go
type Fault struct {
    // HTTP-level (proxy mode)
    HTTPStatus      int               `yaml:"http_status,omitempty"`
    ResponseHeaders map[string]string `yaml:"response_headers,omitempty"`
    ResponseBody    string            `yaml:"response_body,omitempty"`

    // Syscall-level (ebpf mode)
    Errno string `yaml:"errno,omitempty"`  // "ECONNRESET", "EACCES", etc.
}
```

The `Match` struct:

```go
type Match struct {
    // HTTP-level
    Host string `yaml:"host,omitempty"`
    Path string `yaml:"path,omitempty"`

    // Syscall-level
    Syscall string `yaml:"syscall,omitempty"`
}
```

A scenario is invalid if an experiment's `Fault` has both HTTP and syscall
fields, or if `Match` mixes HTTP and syscall fields. Reject at load time
with a clear error.

## 7. CLI surface for v0.1

These are the commands. Flag names are part of the user contract.

```
faultkit run [flags] -- <target> [target args...]
  --scenario <name>          builtin scenario name
  --config <path>            YAML scenario file
  --mode <auto|proxy|ebpf>   force injection mode (default: auto)
  --report <path>            write JSON report to path
  --verbose                  log every fault decision

faultkit doctor
  (no flags)

faultkit scenario list
  (prints builtin + any in $FAULTKIT_HOME)

faultkit scenario show <name>
  (prints loaded YAML)

faultkit version
  --short                    print just the version, no commit/date

faultkit help [command]
  (cobra default)
```

`--scenario` and `--config` are mutually exclusive. If both are missing,
look for `./faultkit.yaml` in cwd. If that's also missing, exit 4
(usage error) with a helpful message.

## 8. Things to flag, not silently work around

If during implementation any of the below comes up, **stop and ask**, do
not invent a solution:

1. **martian's API doesn't expose what you need.** It's actively maintained
   but quiet; if a feature is missing, ask before forking or replacing.
2. **`openai-python` doesn't behave as expected with `HTTPS_PROXY` set.**
   The whole v0.1 demo depends on this; surface immediately.
3. **The kernel version detection is unreliable.** Don't ship a heuristic.
4. **The eBPF verifier rejects `flaky_network.bpf.c`.** This is normal,
   not a sign of a deeper problem — but ask before adding workarounds
   that look like cargo-culting (random `__always_inline`, random bounds
   checks).
5. **Cobra's behavior conflicts with the exit code contract.** The exit
   code constants are load-bearing per `CLAUDE.md`. Don't change them
   to fit cobra; change cobra usage to fit them.
6. **A test requires sudo to run.** Integration tests with eBPF do; that's
   expected. But a unit test requiring sudo is wrong; flag it.
7. **You're tempted to add a new dependency.** Read `CLAUDE.md` again.
   Ask first. The answer is usually "use the stdlib."
8. **Anything that looks like Pro-aware code.** Re-read `CLAUDE.md` §
   "The OSS/Pro boundary." If it's Pro, it doesn't belong in this repo.

## 10. What v0.1 does **not** prove

Be honest with yourself about what shipping v0.1 establishes:

- It proves the architecture works end-to-end. Good.
- It proves you can ship a working CLI. Good.
- It does **not** prove product-market fit, demand, or willingness to
  pay. Those are evaluated at the 6-month checkpoint per `PROJECT_PLAN.md`.
- It does **not** prove the AI-agent positioning is right. The launch
  is the test of that. v0.1 is the precondition for the test, not the
  test itself.
- It does **not** lock you into anything. If v0.2 reveals the proxy is
  the wrong abstraction or the YAML schema is awkward, change them.
  v0.1's job is to be shippable, not perfect.

The strategic anchor remains `PROJECT_PLAN.md`: side project, two weekends
a month, decisions favor shipping over polish, narrow scope over completeness.
When in doubt, do less.
