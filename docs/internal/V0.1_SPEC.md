# faultkit v0.1 implementation spec

**Audience:** Claude Code, working on `github.com/faultkit-dev/faultkit`.
**Status:** active. This document supplements `PROJECT_PLAN.md` and `CLAUDE.md`.
**Scope:** v0.1 only. Anything not listed here is out of scope for this iteration.

If something here conflicts with `CLAUDE.md`, `CLAUDE.md` wins. If something
here conflicts with `PROJECT_PLAN.md`, this document wins (it represents the
current iteration of the plan after a strategic refinement toward AI-agent-first
positioning and a scope expansion to 5 scenarios).

---

## 1. The strategic shift, in one paragraph

The original plan positioned faultkit as an eBPF-first chaos engineering tool
with AI scenarios as a wedge. After review, the positioning was refined:
faultkit is **a fault injection toolkit for AI/agent stacks**, where the
HTTPS proxy is the primary injection mechanism (because LLM/RAG/gateway
failures live in HTTPS semantics on top of TLS), and eBPF is the supporting
mechanism for syscall-level scenarios (backend chaos, future tool-call
scenarios). v0.1 must ship both injection paths working end-to-end with
**three proxy scenarios** and **two eBPF scenarios** — enough to prove the
multi-mode architecture is real and the scenario library has depth, without
crossing into v0.2 territory.

---

## 2. v0.1 deliverables

v0.1 ships when all of the following are true:

1. The five scenarios listed in §3 ship as built-in scenarios (`faultkit
   scenario list` shows all five), all working end-to-end on their target
   platforms.
2. The marquee demo works: `faultkit run --scenario llm-api-degraded --
   pytest tests/agent/` succeeds on macOS and Linux against real
   `openai-python` SDK calls. The proxy returns real HTTP 429s with
   vendor-accurate response bodies. The target test exits non-zero when
   the agent's retry path is broken.
3. The eBPF demo works: `faultkit run --scenario flaky-network --
   <some-target>` succeeds on Linux 5.8+ and demonstrates eBPF-mode
   injection with a real `ECONNRESET`.
4. `faultkit doctor` reports which modes are available on the current
   host and why. Honest, no false positives.
5. Distinct exit codes per `CLAUDE.md` (`0` = pass, `1` = target failed
   under fault, `2` = internal error, `3` = fault never fired, `4` =
   usage).
6. JSON report output via `--report report.json` for CI consumption.
7. GitHub Actions example in `examples/github-actions/` that runs against
   a tiny test target and demonstrates the "fault catches a bug" flow.
8. `make build`, `make test`, `make lint` all pass in CI.
9. README, `docs/scenarios.md`, `docs/quickstart.md`, `docs/ci.md` in
   place. PROJECT_PLAN.md as-is.
10. Apache 2.0 LICENSE file. CONTRIBUTING.md with CLA bot pointer (CLA
    setup itself is out of scope; the doc references it). CODE_OF_CONDUCT.md.

What is **not** in v0.1 (do not build, do not stub, do not scaffold):

- Anthropic, Bedrock, Vertex, or any non-OpenAI vendor fixtures (default
  fallback covers other vendors)
- LD_PRELOAD / DYLD shim mode
- More than two eBPF scenarios
- More than three proxy scenarios
- `tool-call-flaky`, `rag-corruption`, `context-window-squeeze`,
  `disk-full`, `slow-dns` (all v0.2)
- iptables-based redirection (only env-var-based proxy redirection in v0.1)
- BPF cgroup connect hook (deferred to v0.2)
- Pro-aware code of any kind
- Telemetry, analytics, version checks, "phone home"
- Auto-update logic
- Configuration via anything other than YAML files + CLI flags

If a request leans into any of the above, stop and surface it.

## 3. The five scenarios

### 3.1 Proxy scenarios (3)

**1. `llm-api-degraded`** — the marquee. LLM provider returns
429/503/timeout. Vendor-accurate OpenAI error body. Configurable via
`http_status` field. This is the demo featured in the README.

**2. `malformed-json-response`** — LLM returns 200 OK with invalid
JSON in the response body. The proxy mutates the response to inject
the malformation. Targets the failure mode that breaks tool-calling
agents most often in production.

**3. `llm-streaming-cutoff`** — proxy forwards an SSE/streaming
response normally for N tokens, then closes the connection without
sending the `data: [DONE]` terminator. Tests streaming completion
detection in agent code.

### 3.2 eBPF scenarios (2)

**4. `flaky-network`** — `ECONNRESET` on `recvmsg` syscall. Targets
backend chaos and TCP-level resilience. Linux 5.8+ only.

**5. `tool-permission-denied`** — `EACCES` on `openat` syscall.
Targets agent tool-call resilience for filesystem operations. Linux
5.8+ only.

The 60/40 proxy/eBPF split reflects the AI-agent positioning: the
marketing story leads with LLM failures, the eBPF scenarios prove the
multi-mode architecture is real and not just a single-trick pony.

## 4. Architecture decisions, locked

These are not up for re-litigation in implementation. If implementation
reveals a decision is wrong, stop and ask before changing it.

### 4.1 Languages and runtimes

- **CLI, runner, proxy, scenario engine, eBPF loader: Go.** Single binary
  output. `go.mod` Go version: 1.22 minimum.
- **eBPF programs: C with libbpf-style CO-RE.** GPL-2.0 licensed (kernel
  requirement). Compiled with `clang` to `bpf` target, embedded with
  `go:embed` so the released binary needs no clang at runtime.
- **Shim: C.** Not in v0.1, but the directory `shim/` is created empty
  with a placeholder README explaining v0.2 status.

### 4.2 Injection mechanisms

| Mechanism    | Status in v0.1 | Used for                             |
|--------------|----------------|--------------------------------------|
| HTTPS proxy  | shipped        | 3 scenarios (llm-api-degraded, malformed-json-response, llm-streaming-cutoff) |
| eBPF         | shipped        | 2 scenarios (flaky-network, tool-permission-denied) |
| Shim         | not built      | placeholder dir only                  |

The proxy uses **martian v3** (`github.com/google/martian/v3`). This is
the only new dependency v0.1 may add. All other proxy work builds on
stdlib + existing dependencies in `go.mod`.

### 4.3 Proxy redirection: env-var-only in v0.1

v0.1 redirects the target's HTTPS traffic to the proxy via environment
variables (`HTTPS_PROXY`, `HTTP_PROXY`, lowercase variants) and trusts
the per-run CA via `SSL_CERT_FILE`, `REQUESTS_CA_BUNDLE`, `NODE_EXTRA_CA_CERTS`,
`CURL_CA_BUNDLE`. This catches the entire Python/Node OpenAI ecosystem,
which is the v0.1 audience. Clients that ignore `HTTPS_PROXY` (some Go
configurations, some Rust `reqwest` setups) are documented as a known
limitation; eBPF-based redirection is deferred to v0.2.

### 4.4 TLS: per-run ephemeral CA

A fresh CA is generated on every `faultkit run` invocation, written to
a temp file, used for the run, and deleted on `Stop`. Never cache.
Never write to a stable path. Never install in any system trust store.
The CA's `NotAfter` is 24 hours, but the run is expected to clean up
well before that.

### 4.5 OpenAI-only fixtures in v0.1

Vendor-accurate response bodies for OpenAI only. Other vendors fall
through to a generic shape:

```json
{"error": {"message": "...", "type": "rate_limit_error", "code": "rate_limit_exceeded"}}
```

Adding a vendor fixture is a public API commitment per `CLAUDE.md`
rules. Do not add new vendor fixtures in v0.1, even if asked.

### 4.6 The `x-faultkit-synthetic` header

Every synthetic response includes `x-faultkit-synthetic: true`. This is
a deliberate choice: it weakens the "indistinguishable from production"
pitch slightly, but it preserves debuggability and prevents synthetic
responses from being misread by production telemetry if something
leaks. Production agent code does not and should not read this header.
It is human-facing metadata.

### 4.7 No fault decision in eBPF for v0.1

The eBPF programs in v0.1 are dumb: they observe syscalls and inject
based on a userspace-populated map. Fault decisions (probability,
matching, timing) live in Go. This is more verifier-friendly, easier
to test, and keeps the BPF code small. Fault injection itself uses
`bpf_override_return` based on a userspace-populated map. Specifically:

- BPF program checks an LRU hashmap keyed by `(pid, syscall_id)`
- Userspace populates the map ahead of time based on the scenario
- BPF reads the map; if entry exists, calls
  `bpf_override_return(-<errno>)`
- All other logic (probability, host matching, etc.) is in Go

Both eBPF scenarios share this pattern with different syscall hooks
and different errnos.

### 4.8 Streaming-aware proxy

The `llm-streaming-cutoff` scenario requires the proxy to handle
SSE/streaming responses without buffering. The proxy must:

- Detect `Content-Type: text/event-stream` in upstream responses
- Stream bytes through to the client without buffering
- Count tokens (loosely — count `data: ` occurrences in the stream is
  fine for v0.1) and close the connection at the configured threshold
- Not break non-streaming responses (the other proxy scenarios)

Martian supports this; verify with a smoke test before committing
deeply.

### 4.9 Single-shot, foreground only

faultkit runs as `faultkit run -- <target>`. It forks the target, waits
for it to exit, cleans up, and exits with the appropriate code. There
is no daemon mode, no background mode, no `serve` subcommand. This is
the contract from `PROJECT_PLAN.md` and stands.

## 6. Implementation phases

Each phase has explicit success criteria. Do not start phase N+1 until
phase N's criteria are met. If a criterion is unclear, surface it and
ask.

The five-scenario scope adds work in phases 3, 7, and 8 compared to
the original plan. Total budget: ~8-12 weekends.

### Phase 0 — Repo hygiene (~half a weekend)

**Tasks:**
- Initialize the repo structure above (empty/placeholder files where
  needed).
- `go.mod` with declared dependencies: `github.com/cilium/ebpf`,
  `github.com/spf13/cobra`, `github.com/spf13/viper`, `gopkg.in/yaml.v3`,
  `github.com/google/martian/v3`. No others.
- `Makefile` with `build`, `test`, `lint`, `bpf`, `clean` targets.
- `.golangci.yml` with reasonable defaults (`govet`, `staticcheck`,
  `errcheck`, `gofmt`, `goimports`, `gosec`, `revive`).
- LICENSE (Apache 2.0), CONTRIBUTING.md (skeleton), CODE_OF_CONDUCT.md
  (Contributor Covenant 2.1).
- `.github/workflows/ci.yml` running `make lint test` on Linux + macOS.
- Initial commit per `CLAUDE.md` commit conventions.

**Success criteria:**
- `make build` succeeds and produces `bin/faultkit`.
- `bin/faultkit version` prints a version string.
- `make lint test` passes in CI on a fresh clone.

### Phase 1 — Scenario loading and CLI plumbing (~1 weekend)

**Tasks:**
- Define `pkg/scenario.Scenario` and `pkg/scenario.Experiment` structs.
  YAML schema below in §7.
- Implement `pkg/scenario.Load(path string) (*Scenario, error)` and
  `pkg/scenario.LoadBuiltin(name string) (*Scenario, error)`.
- Implement `internal/scenario/builtin/` with all five builtin
  scenarios. Embed their YAML with `go:embed`. Register via `init()`
  per `CLAUDE.md` conventions.
- Implement `cmd/faultkit/main.go` (≤20 LOC of substance, just calls
  `internal/cli.Execute()`).
- Implement `internal/cli/cli.go` with cobra root command and exit
  code dispatch. Define typed exit code constants (`ExitOK`,
  `ExitTargetFailed`, `ExitInternalError`, `ExitFaultNotFired`,
  `ExitUsage`).
- Implement `faultkit version`, `faultkit scenario list`,
  `faultkit scenario show <name>` subcommands.
- Implement skeleton `faultkit run` and `faultkit doctor` (will be
  filled in later phases).

**Success criteria:**
- `faultkit scenario list` prints all five builtin scenarios with
  one-line descriptions.
- `faultkit scenario show <name>` prints the loaded YAML for any of
  the five.
- Loading a malformed YAML fails with a useful error and exit code 4.
- `faultkit version` prints version + commit + build date.
- Unit tests for scenario loading (table-driven, valid + invalid YAMLs
  for each scenario shape).

### Phase 2 — Per-run CA and proxy server (~1 weekend)

**Tasks:**
- Implement `internal/inject/proxy/ca.go`:
  - `NewCA()` generates a per-run ECDSA P-256 root.
  - `MintLeaf(host string)` returns a `*tls.Certificate` for `host`,
    cached per host, with appropriate `DNSNames` SAN.
  - `WriteCertPEM()` writes the CA cert to a temp file and returns
    the path. Tracks the path for cleanup.
  - `Cleanup()` removes the temp file. Idempotent.
- Implement `internal/inject/proxy/server.go`:
  - Constructs a `martian.Proxy` configured with the per-run CA via
    `mitm.NewConfig(ca.cert, ca.key)`.
  - Sets request and response modifiers (the `Faulter`, see Phase 3).
  - Wraps the proxy in an `*http.Server` with sensible timeouts that
    don't break long-running streams (no `ReadTimeout`; bounded
    `ReadHeaderTimeout` only).
- Implement `internal/inject/proxy/proxy.go`:
  - `Injector` struct implementing `inject.Injector`.
  - `Start(ctx, scenario)` generates CA, listens on `127.0.0.1:0`,
    starts the proxy, returns env vars to inject into the target.
  - `Stop(ctx)` shuts down the proxy and cleans up the CA.
  - `Events()` returns the buffered event channel.

**Success criteria:**
- Unit tests for CA generation and leaf minting.
- A manual smoke test where the developer runs the proxy in a
  terminal and points `curl --cacert /tmp/faultkit-ca-*.pem -x
  http://127.0.0.1:8080 https://api.openai.com/v1/models` at it and
  gets a real upstream response.
- Proxy doesn't leak the CA temp file after `Stop()`.

### Phase 3 — Fault matching and three proxy scenarios (~2 weekends)

**Tasks:**

*Phase 3a — matching infrastructure (half a weekend):*
- Implement `internal/inject/proxy/matcher.go`:
  - `Match(*http.Request) *Experiment` returns the first experiment
    whose `match` clause fits, in YAML order.
  - Host matching via `filepath.Match` (supports `*.openai.com`).
  - Path matching same.
- Implement `internal/inject/proxy/faulter.go`:
  - `ModifyRequest`: matches request → checks probability → if firing,
    marks context for synthetic response in `ModifyResponse`.
  - `ModifyResponse`: reads stashed experiment, replaces or mutates
    response per the experiment's `fault` field.
  - Emits `inject.Event` to the event channel for every fault fired.

*Phase 3b — synchronous scenarios: llm-api-degraded + malformed-json-response (half a weekend):*
- Implement `internal/inject/proxy/fixtures/openai.go`:
  - `OpenAIRateLimitResponse(fault Fault)` for `llm-api-degraded`.
    Vendor-accurate JSON shape per OpenAI's docs. Sets
    `x-faultkit-synthetic: true`. For 429: sets `Retry-After`.
  - `OpenAIChatCompletionWithBody(body string)` for
    `malformed-json-response`. Wraps the malformed JSON in a valid
    OpenAI chat completion envelope so the SDK doesn't reject the
    outer structure before reaching the inner content.
- Implement `internal/inject/proxy/fixtures/fixtures.go`:
  - `SyntheticResponse(host string, fault Fault)` dispatches to the
    right vendor based on host. Default: generic shape.

*Phase 3c — streaming scenario: llm-streaming-cutoff (1 weekend):*
- Implement `internal/inject/proxy/streaming.go`:
  - Detects SSE responses (`Content-Type: text/event-stream`).
  - For matched + fault-fired streaming responses: forwards bytes
    through, counting `data:` events, closes the connection without
    `data: [DONE]` after the configured threshold.
  - Critical: must not buffer; bytes must flow to the client as they
    arrive from upstream.
- Update `Faulter.ModifyResponse` to dispatch streaming responses to
  the streaming handler.

**Success criteria:**
- Unit tests for matcher (table-driven, host glob patterns).
- Unit tests for faulter probability gating (deterministic via seeded
  RNG).
- Unit test for OpenAI fixture: response body matches OpenAI's
  documented rate limit error shape exactly.
- Manual smoke test: each of the three proxy scenarios produces the
  expected behavior when run against a real `curl` command targeting
  `api.openai.com`.
- Streaming smoke test: a Python client using
  `openai.OpenAI().chat.completions.create(stream=True)` receives a
  truncated stream and detects the missing `[DONE]` terminator.

### Phase 4 — Runner and target process management (~half a weekend)

**Tasks:**
- Implement `internal/runner/runner.go`:
  - `Runner.Run(ctx, target []string, env []string) (*Result, error)`.
  - Forks the target with merged env (target's env + faultkit-injected env).
  - Pipes stdout/stderr through to faultkit's stdout/stderr (no
    buffering; test output should stream live).
  - Waits for the target to exit. Captures exit code.
  - Propagates SIGINT/SIGTERM from faultkit to the target.
- Implement `internal/runner/errors.go`:
  - `*TargetExitError` typed error containing the target's non-zero
    exit code. The CLI dispatches this to `ExitTargetFailed`.

**Success criteria:**
- Test against a tiny shell script target: `runner.Run("sh", "-c", "exit 7")`
  returns `*TargetExitError` with `ExitCode == 7`.
- Test that env vars passed to `Run` reach the target (target prints
  `os.Getenv("HTTPS_PROXY")`, faultkit asserts on stdout).
- SIGINT to faultkit kills the target and faultkit exits cleanly.

### Phase 5 — `faultkit run` end-to-end with proxy (~half a weekend)

**Tasks:**
- Wire `internal/cli/run.go`:
  1. Parse flags (`--scenario`, `--config`, `--report`, `--mode`, `--`).
  2. Load scenario.
  3. Resolve injection mode. Default selection: if scenario has any
     HTTP experiments → proxy; otherwise → ebpf.
  4. Construct injector (proxy in this phase).
  5. `injector.Start(ctx, scenario)` → get env.
  6. `runner.Run(ctx, target, env)` → get result.
  7. `injector.Stop(ctx)`.
  8. Drain events channel into report.
  9. Write report (terminal always, JSON if `--report`).
  10. Exit with appropriate code.
- Implement `internal/report/terminal.go` and `internal/report/json.go`.

**Success criteria:**
- `faultkit run --scenario llm-api-degraded -- curl https://api.openai.com/v1/models`
  returns a synthetic 429 some percentage of the time, real upstream
  the rest.
- `faultkit run --scenario malformed-json-response -- curl ...`
  returns malformed JSON some percentage of the time.
- `faultkit run --scenario llm-streaming-cutoff -- curl ...` (with a
  streaming endpoint) drops the connection mid-stream some percentage
  of the time.
- `faultkit run --scenario llm-api-degraded -- false` exits with code
  1 and reports "target failed under fault."
- `faultkit run --scenario llm-api-degraded -- true` exits with code
  3 if the target made no HTTPS calls (fault never fired).
- `--report report.json` produces machine-parseable output.

### Phase 6 — `faultkit doctor` and mode resolution (~quarter weekend)

**Tasks:**
- Implement `internal/inject/inject_linux.go` and `inject_other.go`:
  - `AvailableModes() []Mode` returns which modes can run on this
    host.
  - On Linux: check kernel version (5.8+), check `CAP_BPF` /
    `CAP_NET_ADMIN` / root (for eBPF mode), report proxy as always
    available.
  - On non-Linux: proxy always available, eBPF reports unsupported.
- Implement `internal/cli/doctor.go`:
  - Prints platform, kernel version, available modes, why each mode
    is or isn't available.
  - Exits 0 if at least one mode works, 2 otherwise.

**Success criteria:**
- `faultkit doctor` on macOS reports "proxy: ok, ebpf: not supported
  (not Linux)".
- `faultkit doctor` on Linux without `CAP_BPF` reports "proxy: ok,
  ebpf: needs CAP_BPF + CAP_NET_ADMIN or root".
- `faultkit doctor` on Linux with the right caps reports both modes
  ok.

### Phase 7 — eBPF mode and two scenarios (~2 weekends)

**Tasks:**

*Phase 7a — eBPF infrastructure (1 weekend):*
- Write `bpf/common.bpf.h` with shared maps (PID filter, fault config
  map keyed by `(pid, syscall_id)`).
- Write `bpf/flaky_network.bpf.c`:
  - Hooks `tracepoint/syscalls/sys_enter_recvmsg` and the
    corresponding `bpf_override_return` site.
  - Reads from the fault-config map keyed by PID; if a fault is
    configured, calls `bpf_override_return(-ECONNRESET)`.
  - GPL-2.0 SPDX header.
  - Uses CO-RE for any struct field reads.
- Generate Go bindings via `bpf2go` (commit the generated files per
  `PROJECT_PLAN.md` so end users don't need clang).
- Implement `internal/inject/ebpf/ebpf_linux.go`:
  - `Injector` implementing `inject.Injector`.
  - `Start` loads BPF programs, attaches tracepoints, populates the
    config map for the scenario's target PID tree (set after the
    runner forks — thread the target PID through the runner cleanly).
  - `Stop` detaches and unloads.
- Implement `internal/inject/ebpf/ebpf_stub.go` for non-Linux builds
  (returns `ErrUnsupported` from `Start`).

*Phase 7b — second scenario: tool-permission-denied (1 weekend):*
- Write `bpf/tool_perm_denied.bpf.c`:
  - Hooks `tracepoint/syscalls/sys_enter_openat`.
  - Reads from the same shared fault-config map shape; calls

## 7. Scenario YAML schema for v0.1

This is the v0.1 schema. Locked. Additions in v0.2+.

```yaml
# faultkit.yaml — top-level
name: my-scenario              # required, kebab-case
description: |                  # optional, one-line for `scenario list`
  One-line description of what this scenario tests.

# Optional. If absent, no constraint. If present, must match for the
# scenario to be loadable on this platform.
requires:
  platform: linux              # one of: linux, darwin, any
  kernel_min: "5.8"            # only if platform=linux

# One or more experiments. Each is independently evaluated per request
# (proxy mode) or per syscall (ebpf mode). First match wins.
experiments:
  - name: "openai rate-limited"
    fault:
      http_status: 429         # for proxy mode
      response_headers:        # optional, per-vendor sane defaults if absent
        Retry-After: "30"
      response_body: ""        # optional, for malformed-json scenarios; raw body
      stream_cutoff_tokens: 0  # optional, for streaming-cutoff; 0 = no cutoff
    match:
      host: "api.openai.com"   # supports filepath.Match globs
      path: "/v1/*"            # optional, defaults to all paths
    probability: 0.2           # 0.0-1.0; 0.2 = 20% of matching requests
```

For eBPF mode, a different shape:

```yaml
experiments:
  - name: "tcp recv reset"
    fault:
      errno: ECONNRESET        # for ebpf mode
    match:
      syscall: recvmsg         # ebpf mode only
    probability: 0.1
```

The `Fault` struct in `pkg/faulttypes/`:

```go
type Fault struct {
    // HTTP-level (proxy mode)
    HTTPStatus         int               `yaml:"http_status,omitempty"`
    ResponseHeaders    map[string]string `yaml:"response_headers,omitempty"`
    ResponseBody       string            `yaml:"response_body,omitempty"`
    StreamCutoffTokens int               `yaml:"stream_cutoff_tokens,omitempty"`

    // Syscall-level (ebpf mode)
    Errno string `yaml:"errno,omitempty"`  // "ECONNRESET", "EACCES", etc.
}
```

The `Match` struct:

```go
type Match struct {
    // HTTP-level
    Host string `yaml:"host,omitempty"`
    Path string `yaml:"path,omitempty"`

    // Syscall-level
    Syscall string `yaml:"syscall,omitempty"`
}
```

A scenario is invalid if an experiment's `Fault` has both HTTP and
syscall fields, or if `Match` mixes HTTP and syscall fields. Reject
at load time with a clear error.

## 8. CLI surface for v0.1

These are the commands. Flag names are part of the user contract.

```
faultkit run [flags] -- <target> [target args...]
  --scenario <name>          builtin scenario name
  --config <path>            YAML scenario file
  --mode <auto|proxy|ebpf>   force injection mode (default: auto)
  --report <path>            write JSON report to path
  --verbose                  log every fault decision

faultkit doctor
  (no flags)

faultkit scenario list
  (prints builtin + any in $FAULTKIT_HOME)

faultkit scenario show <name>
  (prints loaded YAML)

faultkit version
  --short                    print just the version, no commit/date

faultkit help [command]
  (cobra default)
```

`--scenario` and `--config` are mutually exclusive. If both are
missing, look for `./faultkit.yaml` in cwd. If that's also missing,
exit 4 (usage error) with a helpful message.

## 9. Things to flag, not silently work around

If during implementation any of the below comes up, **stop and ask**,
do not invent a solution:

1. **martian's API doesn't expose what you need** for streaming. SSE
   handling is the most likely place this happens. If a feature is
   missing, ask before forking or replacing.
2. **`openai-python` doesn't behave as expected with `HTTPS_PROXY` set.**
   The whole v0.1 demo depends on this; surface immediately.
3. **The streaming cutoff doesn't behave consistently.** SSE has many
   shapes; if you hit a case where the proxy mishandles it, surface
   before adding workarounds.
4. **The kernel version detection is unreliable.** Don't ship a heuristic.
5. **The eBPF verifier rejects either BPF program.** This is normal,
   not a sign of a deeper problem — but ask before adding workarounds
   that look like cargo-culting.
6. **The two BPF programs conflict** (e.g., overlapping maps,
   conflicting tracepoints). The shared map design should prevent
   this; if it doesn't, surface.
7. **Cobra's behavior conflicts with the exit code contract.** The
   exit code constants are load-bearing per `CLAUDE.md`. Don't change
   them to fit cobra; change cobra usage to fit them.
8. **A test requires sudo to run.** Integration tests with eBPF do;
   that's expected. But a unit test requiring sudo is wrong; flag it.
9. **You're tempted to add a new dependency.** Read `CLAUDE.md` again.
   Ask first. The answer is usually "use the stdlib."
10. **Anything that looks like Pro-aware code.** Re-read `CLAUDE.md` §
    "The OSS/Pro boundary." If it's Pro, it doesn't belong in this repo.

## 11. Time budget honesty

The five-scenario scope adds ~3-5 weekends to the original 5-7
weekend estimate. Total: **8-12 weekends**, or 4-6 months at a
two-weekends-a-month cadence.

Where the extra time goes:

- Phase 3 expansion (3 proxy scenarios instead of 1): +1 weekend
- Phase 7 expansion (2 eBPF scenarios instead of 1): +1 weekend
- Phase 8 expansion (3 integration tests instead of 1): +0.5 weekends
- Streaming-cutoff complexity (SSE handling is bug-prone): +0.5-1.5

   marketing

The cuts NOT to make under time pressure: the proxy itself, the OpenAI
fixture, the integration test against real openai-python, the doctor
command, the exit code contract. Those are the credibility-establishing
pieces.

## 12. What v0.1 does **not** prove

Be honest with yourself about what shipping v0.1 establishes:

- It proves the architecture works end-to-end. Good.
- It proves you can ship a working CLI. Good.
- It proves the multi-mode design is real (5 scenarios across 2 modes).
  Better than the 2-scenario version proved.
- It does **not** prove product-market fit, demand, or willingness to
  pay. Those are evaluated at the 6-month checkpoint per
  `PROJECT_PLAN.md`.
- It does **not** prove the AI-agent positioning is right. The launch
  is the test of that. v0.1 is the precondition for the test, not the
  test itself.
- It does **not** lock you into anything. If v0.2 reveals the proxy is
  the wrong abstraction or the YAML schema is awkward, change them.
  v0.1's job is to be shippable, not perfect.

The strategic anchor remains `PROJECT_PLAN.md`: side project,
sustainable cadence, decisions favor shipping over polish, narrow
scope over completeness. When in doubt, do less.
